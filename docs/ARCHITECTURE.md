# Архитектура проекта

Документ объясняет, как связаны файлы проекта и почему код разделен именно так.

## Общая схема

```text
lib/main.dart
  создает приложение, тему, маршруты и Provider

lib/ui/
  показывает интерфейс и вызывает действия пользователя

lib/state/schedule_controller.dart
  хранит состояние и связывает UI с API

lib/services/ruz_api_client.dart
  выполняет HTTP-запросы и проверяет ответы

lib/models/
  превращает JSON из API в понятные Dart-объекты
```

Главный принцип: каждый слой отвечает только за свою задачу.

- UI не знает точные URL API.
- API-клиент не знает, как выглядят экраны.
- Модели не делают запросы.
- Контроллер хранит состояние и управляет загрузкой.

## Поток поиска

1. Пользователь вводит текст в `TargetSearchPanel`.
2. Виджет ждет 450 мс debounce-паузы.
3. `TargetSearchPanel` вызывает `ScheduleController.search(term)`.
4. Контроллер проверяет длину строки и ставит `searchStatus = loading`.
5. Контроллер вызывает `RuzApiClient.searchTargets(...)`.
6. API-клиент отправляет `GET /api/search`.
7. JSON превращается в `ScheduleTarget`.
8. Контроллер сохраняет `searchResults`.
9. UI автоматически перерисовывает список.

## Поток открытия расписания

1. Пользователь нажимает на результат поиска.
2. `HomeScreen._openSchedule()` открывает маршрут `/schedule/:kind/:id`.
3. `go_router` создает `ScheduleScreen`.
4. `ScheduleScreen` после первого кадра вызывает `loadScheduleFor(target)`.
5. Контроллер вызывает `RuzApiClient.loadSchedule(...)`.
6. API-клиент отправляет запрос к расписанию.
7. JSON превращается в список `ScheduleLesson`.
8. Контроллер сохраняет `lessons`.
9. `ScheduleWeekGrid` раскладывает занятия по дням.

## Почему используется Provider

`provider` нужен, чтобы не передавать данные вручную через много конструкторов.

В `main.dart` создается:

```dart
ChangeNotifierProvider(
  create: (_) => ScheduleController(apiClient: apiClient),
  child: const FaScheduleApp(),
)
```

После этого любой экран может:

- читать состояние: `context.watch<ScheduleController>()`;
- вызвать действие: `context.read<ScheduleController>().search(value)`.

## Почему используется go_router

`go_router` решает задачу переходов между экранами.

Маршруты:

- `/` — экран поиска;
- `/schedule/:kind/:id` — экран расписания.

В URL передаются `kind` и `id`, а полный объект `ScheduleTarget` передается через `extra`. Это удобно: экран расписания сразу получает название и описание выбранной группы/преподавателя.

## Основные файлы

### `lib/main.dart`

Точка входа. Здесь:

- инициализируется русская локализация дат;
- создается `RuzApiClient`;
- создается `ScheduleController`;
- подключается `Provider`;
- настраивается тема;
- задаются маршруты.

### `lib/state/schedule_controller.dart`

Центральное состояние. Хранит:

- тип поиска: группа или преподаватель;
- результаты поиска;
- выбранный объект;
- текущую неделю;
- список занятий;
- статусы загрузки;
- сообщение ошибки;
- включен ли прокси.

### `lib/services/ruz_api_client.dart`

Сетевой слой. Только этот файл знает:

- базовый URL RUZ;
- URL локального прокси;
- endpoint поиска;
- endpoint расписания;
- какие query-параметры нужны API.

### `lib/models/schedule_target.dart`

Модель результата поиска. Нормализует группу/преподавателя в единый объект:

- `id`;
- `title`;
- `subtitle`;
- `kind`.

### `lib/models/schedule_lesson.dart`

Модель занятия. Забирает из большого JSON только нужные поля:

- дата;
- время;
- дисциплина;
- тип занятия;
- аудитория;
- корпус;
- преподаватель;
- группа;
- примечание.

### `lib/ui/screens/home_screen.dart`

Главный экран поиска. Отвечает за компоновку:

- на широком экране слева поиск, справа подсказка;
- на узком экране только поиск.

### `lib/ui/screens/schedule_screen.dart`

Экран расписания. Отвечает за:

- загрузку расписания при открытии;
- кнопки переключения недель;
- шапку с метриками;
- отображение календаря и списка занятий.

## Почему дизайн отличается от референса

Проект сделан как отдельное Windows-приложение:

- широкая компоновка под desktop;
- боковая панель поиска;
- недельная сетка;
- карточки занятий;
- отдельная панель настроек API;
- аккуратные состояния загрузки и ошибок.
